Cache performance · C++

Why this loop is 5× slower - same logic, different memory order

Two functions, same matrix, same number of additions. One takes ~50 ms, the other ~250 ms. The difference has nothing to do with the algorithm, it's about how memory is accessed.

The code

Version A: row-major traversal (cache-friendly) 
void traverseRowMajor()
{
    double sum = 0.0;
    for (int row = 0; row < MATRIX_SIZE; ++row)
        for (int col = 0; col < MATRIX_SIZE; ++col)
            sum += matrix[row][col];  // walks memory in order
}
Version B: column-major traversal (cache-unfriendly) 
void traverseColumnMajor()
{
    double sum = 0.0;
    for (int col = 0; col < MATRIX_SIZE; ++col)
        for (int row = 0; row < MATRIX_SIZE; ++row)
            sum += matrix[row][col];  // jumps forward in memory each step
}

The only difference is the loop order. Let's look at what the hardware sees.

What the cache sees

C++ stores 2D arrays in row-major order. All of row 0 first, then row 1, then row 2. A cache line is 64 bytes, which holds 8 doubles. So fetching matrix[0][0] automatically loads matrix[0][1] through matrix[0][7] into the cache for free.

Both versions execute exactly the same number of floating-point additions. The CPU is not doing less work in version A, it's doing the same work without waiting for RAM on every single memory access.

Why this happens

● Row-major: riding the cache line

When you access matrix[row][col], the CPU fetches a full 64-byte cache line from RAM - that's 8 doubles in a row. The next 7 iterations access those same bytes, already in one of the caches (latency: ~1 ns). You pay the RAM cost once and amortize it over 8 operations.

● Column-major: RAM access latency on every step

Accessing matrix[row][col] with the column as the outer loop means the next access is matrix[row+1][col], which is MATRIX_SIZE×8 bytes away. For example, in a 4096x4096 matrix, this is 32,768 bytes apart. That's a completely different cache line. The one you just loaded gets evicted unused. Every access costs a full RAM round-trip.

The takeaway

The hardware doesn't fetch single values - it fetches 64-byte cache lines. Code that accesses memory in the order it was laid out gets 8 values for the price of 1. Code that skips around pays full price every time.

This is why knowing the data layout matters as much as knowing the algorithm. A 5× slower code might happen every time if memory access patterns are ignored.

Rule of thumb: in C/C++, always iterate over the last (rightmost) index in the inner loop. That's the direction memory is stored.

Try it yourself

Download RowColumnMajor.cpp. This file has the full code with time measurements using std::chrono::high_resolution_clock. Compile and run it. The numbers will vary by machine but the ratio should hold.

Try changing MATRIX_SIZE from 4096 to 512 and watch the gap shrink. Smaller matrices fit more easily in L2/L3 cache, so the miss penalty is less severe. That's the cache hierarchy in action.
